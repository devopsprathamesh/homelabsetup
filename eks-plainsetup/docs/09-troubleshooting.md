# 09. Troubleshooting

| Symptom | Likely cause | Where to look |
|---|---|---|
| `create-nodegroup` fails immediately | Node role missing a required managed policy, or subnets don't have enough free IPs | `aws eks describe-nodegroup --cluster-name $CLUSTER_NAME --nodegroup-name core-ng --query 'nodegroup.health'` |
| Nodes join but stay `NotReady` | VPC CNI pods (`aws-node`) crashlooping â€” usually the Pod Identity association for `kube-system/aws-node` wasn't created before the addon ([05](05-pod-identity-core-addons.md)) | `kubectl -n kube-system logs ds/aws-node`; re-check `aws eks list-pod-identity-associations --cluster-name $CLUSTER_NAME` |
| `AccessDenied` in any controller's logs (cluster-autoscaler, Karpenter, ALB controller) | Pod Identity association points at the wrong service account name/namespace, or the IAM policy wasn't attached before the pod started | Confirm `kubectl get sa <name> -n <namespace>` matches the association exactly; restart the pod after fixing (`kubectl delete pod ...`) â€” credentials are fetched at pod start, a running pod won't pick up a fixed association on its own |
| PVC stuck `Pending` | `aws-ebs-csi-driver` addon not `ACTIVE`, or its Pod Identity association missing | `kubectl -n kube-system get pods -l app=ebs-csi-controller`, `kubectl describe pvc <name>` |
| `Service type=LoadBalancer` never gets an `EXTERNAL-IP` | ALB controller not running, or subnets missing the `kubernetes.io/role/elb` / `internal-elb` tags from [02](02-networking-vpc.md) | `kubectl -n kube-system logs deploy/aws-load-balancer-controller`; re-check subnet tags with `aws ec2 describe-subnets --subnet-ids ${PUBLIC_SUBNET_IDS[@]}` |
| `aws eks create-addon --resolve-conflicts OVERWRITE` still errors on conflict | A previous manual edit to the self-managed manifest changed a field EKS won't silently overwrite | Retry with `--resolve-conflicts OVERWRITE` (not `NONE`); as a last resort delete the conflicting resource and let the addon recreate it |
| **Cluster Autoscaler** never scales up despite pending pods | ASG tags from [06/A1](06-node-autoscaling.md#a1-tag-the-node-groups-auto-scaling-group-for-discovery) missing, or `maxSize` already reached | `aws autoscaling describe-tags --filters "Name=key,Values=k8s.io/cluster-autoscaler/enabled"`; `kubectl -n cluster-autoscaler logs deploy/cluster-autoscaler \| grep -i "no scale"` |
| **Karpenter** never provisions nodes | Controller Pod Identity association missing/wrong, or `EC2NodeClass`/`NodePool` subnet/security-group selector tags don't match anything | `kubectl -n kube-system logs deploy/karpenter \| tail -50`; `kubectl describe ec2nodeclass default` â€” check the `Ready` condition |
| **Karpenter** launches nodes that never join the cluster | `role:` in `EC2NodeClass` doesn't match `${CLUSTER_NAME}-node-role`, or that role is missing `AmazonEKSWorkerNodePolicy` | `kubectl get nodeclaims`, then check the underlying EC2 instance's system log in the console; confirm `aws iam list-attached-role-policies --role-name ${CLUSTER_NAME}-node-role` |
| **Karpenter** doesn't terminate spot instances gracefully before interruption | SQS queue or EventBridge rules from [06/B2](06-node-autoscaling.md#b2-sqs-interruption-queue--eventbridge-rules) missing, or `settings.interruptionQueue` in the Helm install doesn't match the actual queue name | `aws sqs get-queue-attributes --queue-url $QUEUE_URL --attribute-names All`; `aws events list-rules --name-prefix ${CLUSTER_NAME}-karpenter` |
